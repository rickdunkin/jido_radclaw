## Fix 1 (P2) — Preserve usage on certificate runner failures

**File:** `lib/jido_claw/tools/verify_certificate.ex`, `execute_cert/2`, lines 174-176.

**Bug.** JidoAI's `RunStrategy` error path (`deps/jido_ai/lib/jido_ai/actions/reasoning/run_strategy.ex:263-287`) returns `{:error, %{strategy, status, output, usage, diagnostics}}` — usage is present on failure. The current clause replaces it with `%{}`:

```elixir
{:error, reason} ->
  {:error, %{reason: "Reasoning strategy failed: #{inspect(reason)}", usage: %{}}}
```

So `Telemetry.extract_tokens/1` sees `usage: %{}` and writes a row with `tokens_in: nil`, `tokens_out: nil` — the same drop we fixed in P1 for parse-failure rows. Cert runner failures still lose accounting.

**Change.** Extract usage from map-shaped errors before formatting, with string/atom key fallback to match the tolerant style in `Telemetry.tokens_from_usage/1`:

```elixir
{:error, reason} ->
  usage =
    if is_map(reason) do
      Map.get(reason, :usage) || Map.get(reason, "usage") || %{}
    else
      %{}
    end

  {:error, %{reason: "Reasoning strategy failed: #{inspect(reason)}", usage: usage}}
```

One call site, so inline rather than a helper.

**User-facing behavior is unchanged.** The outer `run/2` error clause at line 101-102 matches `{:error, reason} when is_binary(reason)` and surfaces the formatted string — same as today. Only the telemetry row gets richer.

**Test.** Add to `test/jido_claw/tools/verify_certificate_test.exs` under the existing "reasoning telemetry integration" describe block:

```elixir
defmodule FailWithUsageRunner do
  @moduledoc false

  def run(_params, _context) do
    {:error,
     %{
       strategy: :cot,
       status: :failure,
       output: nil,
       usage: %{input_tokens: 77, output_tokens: 11},
       diagnostics: %{}
     }}
  end
end

test "persists runner-error token usage on runner failure" do
  params = %{code: "code", specification: "spec"}
  context = %{reasoning_runner: FailWithUsageRunner}

  assert {:error, msg} = VerifyCertificate.run(params, context)
  assert msg =~ "Reasoning strategy failed"

  row = find_cert_row_by_tokens(77, 11)
  assert row
  assert row.status == :error
  assert row.execution_kind == :certificate_verification
end
```

The existing `FailRunner` (line 36-42) returns a bare-string reason — keep it for the user-facing-message assertion and add the new one above for the map-shaped error path. Both scenarios exist in production (bare-string exits from OTP, map payloads from JidoAI RunStrategy).

## Fix 2 (P3) — Single source of truth for tool list in `agent.ex`

**File:** `lib/jido_claw/agent/agent.ex`, lines 6-44 (inline `tools:`) + 53-83 (`@tool_modules`).

**Bug.** Two hand-maintained lists. The original plan (`.claude/plans/please-put-together-a-serene-knuth.md:243-256`) called for a single `@tool_modules` attribute referenced in both `use Jido.AI.Agent` and `tool_modules/0`. Implementation diverged — both lists match today but the next tool addition can desync them silently.

**Why the obvious refactor doesn't work.** `Jido.AI.Agent` explicitly rejects module attributes in `tools:` at `deps/jido_ai/lib/jido_ai/agent.ex:168-174` — the macro pattern-matches `{:@, meta, [{name, _, _}]}` and raises `CompileError` with "Module attributes (@name) are not supported in tool_context, tools, or specialists. Define the value inline or use a compile-time constant." So `tools: @tool_modules` fails at compile time; the attribute is passed as AST, not expanded to a literal list.

**Change.** Keep the literal list inline in `use Jido.AI.Agent`, and derive `tool_modules/0` from the `strategy_opts/0` function that `use Jido.Agent` generates (via `use Jido.AI.Agent`). `strategy_opts/0` is defined at `deps/jido/lib/jido/agent.ex:713-719` and returns a keyword list that includes `:tools` — populated from the exact list the `use` macro accepted, resolved to atoms.

Delete the duplicate `@tool_modules` attribute. Rewrite `tool_modules/0`:


Net result: one literal list (the `use` macro's `tools:` argument), and `tool_modules/0` reads from it at runtime. Adding a tool means editing one place.

If the compile succeeds and the REPL starts, the derivation works. ## Fix 3 (P3) — Runner injection in `Reason.run_runner_strategy/5`

**File:** `lib/jido_claw/tools/reason.ex`, `run_runner_strategy/5`, line 110-144.

**Bug.** Line 129 hardcodes `Jido.AI.Actions.Reasoning.RunStrategy.run/2`, so the `reasoning_runner: CotRunner` context passed by `reason_test.exs:78` is silently ignored. The test asserts only the telemetry row shape — which gets written regardless of dispatch outcome, so a regression in alias-to-base dispatch (e.g., base_atom wrong, runner called with a stale strategy name) wouldn't fail the test reliably. `VerifyCertificate` (line 118) and `RunPipeline` (line 87) both use the `runner = Map.get(context, :reasoning_runner, Jido.AI.Actions.Reasoning.RunStrategy)` pattern.

**Change.** Thread the runner through `run_runner_strategy/5`:

```elixir
defp run_runner_strategy(strategy_name, base_atom, base_name, prompt, context) do
  workspace_id = get_in(context, [:tool_context, :workspace_id])
  project_dir = get_in(context, [:tool_context, :project_dir])
  runner = Map.get(context, :reasoning_runner, Jido.AI.Actions.Reasoning.RunStrategy)

  run_params = %{
    strategy: base_atom,
    prompt: prompt,
    timeout: 60_000
  }

  Telemetry.with_outcome(
    strategy_name,
    prompt,
    [
      execution_kind: :strategy_run,
      base_strategy: base_name,
      workspace_id: workspace_id,
      project_dir: project_dir
    ],
    fn -> runner.run(run_params, %{}) end
  )
  |> case do
    {:ok, result} ->
      {:ok,
       %{
         strategy: strategy_name,
         output: extract_output(result),
         status: Map.get(result, :status),
         usage: Map.get(result, :usage, %{})
       }}

    {:error, reason} ->
      {:error, format_error(strategy_name, reason)}
  end
end
```

Only one line changes materially: `fn -> Jido.AI.Actions.Reasoning.RunStrategy.run(run_params, %{}) end` → `fn -> runner.run(run_params, %{}) end`, plus the `runner =` lookup two lines up.

**Test updates in `test/jido_claw/tools/reason_test.exs`.**

(a) Change `CotRunner` to reflect the incoming `strategy` parameter in its output. Without this, the runner returns a fixed payload regardless of params, and the test proves *injection happened* but not *that the resolved base atom was passed correctly*. A regression where `run_runner_strategy/5` resolves `"fast_reviewer"` to, say, `:got` instead of `:cot` would still see `CotRunner` run and emit its fixed output — test passes, dispatch is broken.

Replace `CotRunner` (lines 14-24) with:

```elixir
defmodule CotRunner do
  @moduledoc false

  # Reflects the incoming strategy atom in :output so tests can assert the
  # resolved base was dispatched correctly (not just that *some* runner ran).
  def run(%{strategy: strategy}, _context) do
    {:ok,
     %{
       output: "ran with strategy=#{inspect(strategy)}",
       usage: %{input_tokens: 11, output_tokens: 22}
     }}
  end
end
```

(b) Tighten the "direct 'cot' writes a :strategy_run row" test (lines 74-89). Assert the injected runner actually ran with the resolved `:cot` atom:

```elixir
test "direct 'cot' dispatches to the injected runner with base :cot and writes a :strategy_run row" do
  assert {:ok, result} =
           Reason.run(
             %{strategy: "cot", prompt: "some cot reasoning task"},
             %{reasoning_runner: CotRunner}
           )

  assert result.strategy == "cot"
  assert result.output == "ran with strategy=:cot"
  assert result.usage == %{input_tokens: 11, output_tokens: 22}

  row = find_row(strategy: "cot", execution_kind: :strategy_run)
  assert row
  assert row.base_strategy == "cot"
  assert row.tokens_in == 11
  assert row.tokens_out == 22
  assert row.status == :ok
end
```

(c) Tighten the "non-react alias" test (lines 112-130) — this is the one that actually proves alias-to-base dispatch. Pass `CotRunner` via context and assert the runner received `:cot`, not `:fast_reviewer`:

```elixir
test "non-react alias resolves to base :cot when dispatching to the runner" do
  with_user_strategy(
    """
    name: fast_reviewer
    base: cot
    """,
    fn ->
      assert {:ok, result} =
               Reason.run(
                 %{strategy: "fast_reviewer", prompt: "review this neutral bucket placeholder"},
                 %{reasoning_runner: CotRunner}
               )

      # User-facing name preserved in the result.
      assert result.strategy == "fast_reviewer"
      # But the runner was invoked with the resolved base atom :cot, not :fast_reviewer.
      assert result.output == "ran with strategy=:cot"

      row = find_row(strategy: "fast_reviewer", execution_kind: :strategy_run)
      assert row
      assert row.base_strategy == "cot"
    end
  )
end
```

Together (b) and (c) pin the dispatch contract: user name preserved in telemetry and return value, resolved base atom passed to the runner.
